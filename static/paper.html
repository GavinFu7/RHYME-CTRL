RHYME CTRL — Rap Rhyme Highlighter

A fully local system for phonetic rhyme analysis, word-timed alignment, and cinematic lyric visualization.

Abstract

This document describes RHYME CTRL, a fully-local computational pipeline for the phonetic, structural, and temporal analysis of rap lyrics. The system integrates sequence-to-sequence acoustic alignment (via Whisper), a vowel-centric rhyme-family detector, locality-aware rhyme grouping, interactive finetuning, and a cinematic HTML renderer driven by headless browser capture into a single architecture capable of producing synchronized, rhyme-colored lyric videos directly from MP3 audio.

The innovation lies in: (1) a three-tier representation of rhyme structure (word-level, sub-word-level, phoneme-tail), (2) a locality-aware rhyme grouping framework implemented in rhyme_core.py, (3) a Whisper-based alignment-to-lyrics synchronization engine implemented in auto_align.py, and (4) a cinematic HTML rendering pipeline (capture.html) controlled via window.capturePlayer and orchestrated by Playwright (capture_video.py).

We formalize rhyme matching as a phoneme-tail similarity problem under a weighted metric and show how the implemented heuristics approximate this objective. We additionally outline a probabilistic HMM/CRF interpretation of the rhyme-labeling problem as future work.

1. Introduction

Rap exhibits dense, multi-layered rhyme structure: internal rhymes, multisyllabic chains, cross-line patterns, assonance, consonance, and deliberate phonetic distortions. Unlike traditional poetry, rhyme is not restricted to line-final suffixes; instead, sequences like foolishly / cluelessly / buffoonishly form phonetic clusters whose rhyme-bearing segments can appear mid-word and span multiple bars.

Classical rhyme detection methods—Levenshtein distance on orthographic suffixes, dictionary-based rhyme lists, or CMUdict end-rhyme detectors— fail to capture the rap-specific structure of phonetic recurrence. They typically ignore performance timing, treat rhyme as a static property of isolated lines, and do not support multi-line rhyme visualization.

RHYME CTRL aims to operationalize a mathematically grounded, performance-aware model of rhyme structure: one that respects phoneme tails centered on stressed vowels, accounts for temporal locality in rhyme grouping, and ultimately produces visualizations that match a human listener’s perception of rhyme schemes across bars and verses.

2. System Overview

RHYME CTRL’s pipeline can be summarized as:

  • Transcription & alignment: Whisper transcribes audio and provides word-level timestamps.
  • Lyrics alignment: user-supplied lyrics are aligned to ASR tokens via a DP-like alignment (auto_align.py).
  • Rhyme analysis: each lyric token is mapped to a phoneme sequence, a rhyme tail, and a locality-aware rhyme group (rhyme_core.py).
  • Interactive views: text mode, auto-align mode, performance mode, and finetune mode, all implemented as HTML templates.
  • Cinematic capture: a fixed 1280×720 capture view (capture.html) driven by window.capturePlayer.
  • Video export: capture_video.py orchestrates browser capture via Playwright and merges frames with audio via ffmpeg.

3. Mathematical Model of Rhyme

3.1 Phoneme-Tail Extraction (Implemented)

For a token \( w \), we let phones_for_word(w) return a phonemic transcription: 

φ(w) = (p₁, p₂, …, pₖ)

In rhyme_core.py, the function last_stressed_vowel_index(phones) finds the last stressed vowel. The tail operator \( T(w) \) used by the code is: 

T(w) = (p_j, p_{j+1}, …, p_k)

where \( p_j \) is the last stressed vowel. This is implemented by rhyme_tail_phones(phones) and then stored into the token as token["rhyme_tail"].

3.2 Rhyme Similarity (Implemented)

The rhyme similarity function is implemented as rhyme_similarity(p1, p2) in rhyme_core.py. It uses:

  • Vowel nucleus identity (via last_stressed_vowel)
  • Common suffix of base phones in rhyme_tail_phones
  • Optional head-rhyme matches (for multi-syllable patterns)

Conceptually, this corresponds to: 

score(wᵢ, wⱼ) ≈ vowel_score(T(wᵢ), T(wⱼ)) + tail_score(T(wᵢ), T(wⱼ))

The code uses simple floats (0.0–1.0), but the paper abstracts this as a similarity metric over phoneme-tails.

3.3 Locality-Aware Grouping (Implemented)

Rhyme grouping is implemented in group_rhymes(lines_tokens, threshold). Each token is assigned a global position pos and each group stores a list of positions. The effective score per group is: 

effective = rhyme_similarity(phones, proto) - α * locality_penalty(pos, positions)

where locality_penalty returns a normalized distance to the nearest existing occurrence in that group. This is the concrete version of: 

eff(wₜ, g) = sim(T(wₜ), C_g) - λ · dist(t, μ_g)

Groups are chosen by maximizing effective, or a new group is created if effective < threshold.

3.4 Probabilistic Interpretation (Not Yet Implemented)

The HMM/CRF model described in the paper (with hidden rhyme labels \(z_t\) and potentials over tails and local neighborhoods) is not yet implemented. It is a formalization of what the greedy grouping is approximating: 

p(z₁:T | tails) ∝ ∏ₜ exp(ψₜ(zₜ) + ψₜ,ₜ₊₁(zₜ, zₜ₊₁))

The current code relies on a fast greedy assignment rather than Viterbi/CRF inference.

4. Alignment Engine

The alignment pipeline lives in auto_align.py. It does:

  1. Whisper transcription with word_timestamps=True.
  2. ASR word sequence via build_asr_word_sequence.
  3. Lyrics word sequence via build_lyrics_word_sequence.
  4. Alignment via align_word_sequences using difflib.SequenceMatcher.

SequenceMatcher approximates the dynamic programming alignment used in classical ASR, giving a mapping from each lyric word index to an ASR word index (or None if unaligned). This mapping is then used to copy Whisper’s timestamps onto the lyric tokens.

5. Cinematic Renderer

The cinematic renderer is implemented in capture.html. It:

  • Accepts audio path + aligned entries from the backend.
  • Constructs a fixed-size (1280×720) DOM with rhyme-colored tokens.
  • Implements smooth vertical scroll tied to the current time.
  • Assigns a glow and focus state to the active word.
  • Exposes window.capturePlayer with: setTime(t), getDuration(), and update().

The capture_video.py script drives this by repeatedly calling capturePlayer.setTime(t), taking screenshots with Playwright, and then passing the PNG sequence to ffmpeg to build the final MP4.

6. Related Work

Text-Only Rhyme Analyzers. Many systems operate purely over text, using CMUdict and deterministic suffix rules to identify rhyme families and measure rhyme density. They do not incorporate performance timing or treat rhyme as a time-dependent phenomenon. RHYME CTRL differs in that it anchors rhyme in time via word timestamps.

Lyric Synchronization (Karaoke Systems). Classical approaches use HMMs or DTW-based forced alignment between audio features and phoneme sequences. These systems produce word timings but typically do not model rhyme structure or offer rich visual semantics. RHYME CTRL builds on similar alignment primitives but layers a rhyme-centric visual semantics on top.

Rap Rhyme Density Tools. Tools like Raplyzer or DeepRhyme evaluate rhyme density and internal rhymes from text alone. They treat rhyme as a static property of a lyric snippet. RHYME CTRL extends this paradigm to continuous time and multi-view visualization (text, perform, capture), providing a time-synchronous view of rhyme patterns.

Poetry Visualization. Static visualizations of rhyme schemes, stress patterns, or prosodic structures exist in the literature, but rarely as cinematic, time-synchronized videos. RHYME CTRL contributes a full-stack approach from raw audio to stylized video exports.

7. Complexity Analysis

Let:

  • \(N\): number of lyric tokens (words)
  • \(S\): number of ASR tokens (Whisper outputs)
  • \(G\): number of rhyme groups
  • \(T\): audio duration in seconds
  • \(\text{fps}\): frames per second for video capture
  • \(F = T \cdot \text{fps}\): total frames

7.1 Whisper Transcription

Whisper’s transformer model dominates runtime. Complexity is roughly: 

O(AudioLength × d²)

for model dimension d. We treat this as a black-box cost.

7.2 ASR ↔ Lyrics Alignment

align_word_sequences uses difflib.SequenceMatcher on sequences of length N and S. Worst-case: 

O(N × S)

7.3 Rhyme Grouping

For each token:

  • Phoneme lookup: O(1)
  • Tail similarity: O(L) with small constant tail length L
  • Compare to all groups: O(G) per token

Total: 

O(N × G)

7.4 HTML Rendering

In /perform, each update toggles classes for one line and one word, plus a scroll. This is effectively O(1) per frame.

7.5 Cinematic Capture

For cinematic capture: 

F = T × fps

For each frame:

  • JS update via setTime(t): O(1)
  • Screenshot: O(W × H) pixels (fixed at 1280×720)

ffmpeg encoding similarly scales with O(F × frame_size).

7.6 Summary

Ignoring Whisper’s model cost, the overall complexity is: 

O(N × S) + O(N × G) + O(F × W × H)

which is linear in text size and video length for realistic song-scale inputs.

8. Conclusion

RHYME CTRL combines DSP, NLP, computational phonology, alignment, and cinematic rendering into a cohesive system for rap analysis. It treats rhyme as a time-synchronous, vowel-centric phenomenon and integrates it into a multi-view interface: text mode, auto-align mode, performance mode, finetune mode, and a dedicated cinematic capture mode.

The implemented rhyme-tail and locality-aware grouping provide a practical approximation to a more general HMM/CRF rhyme-labeling model, and the HTML-based renderer offers a flexible, high-fidelity medium for producing publishable lyric videos. The structure of the code and the abstractions introduced here make RHYME CTRL both a production-ready tool and a research platform for future work in rap rhyme modeling, flow analysis, and prosodic visualization.
